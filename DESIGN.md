## Problem Statement
The existing `pomo` application (`github.com/recoskyler/pomo`) is a robust, cross-platform Pomodoro focus timer with multi-window desktop support (`desktop_multi_window`) and custom RGB webhooks. However, focus timers only capture active sprint intervals, leaving the rest of the 24-hour daily cycle untracked. To maintain complete visibility into how time is spent across all daily activities without introducing tracking friction, we need to encapsulate `pomo` inside a unified **Time Tracker** application package. This system must trigger hourly reminders (with sound alerts) on Android 16 (`API 36`) and macOS, allowing one-tap logging of pre-defined or dynamically created tags directly to a Notion PARA dashboard (`time logging reports`), while providing multi-period stacked charts (`fl_chart`) and actionable missed-time gap tracking that automatically respects designated sleep hours.

## What Makes This Cool
1. **Instant 1-Tap Check-In via Persistent Android 16 Notification:** Instead of forcing you to unlock your phone and navigate through menus every hour, the app maintains a lightweight, persistent ongoing notification on Android 16 (`API 36`) with live action buttons (`[ Log 60m Work ]`, `[ Switch Tag ]`, `[ Open Grid ]`). When the top of the hour chimes (`pop.aac`), one tap on the notification shade instantly logs the hour directly to your Notion PARA dashboard.
2. **Vibrant 4x5 Card Grid with Celebratory Micro-Animations:** When opening the `Time Tracker` tab, you are greeted by an ultra-fast, visually stunning 4x5 grid of colorful rounded-square activity cards (`Games`, `Read`, `Work`, `Exercise`, `Sleep`, etc.). Tapping any card logs the interval, triggers a celebratory UI burst, and plays a crisp sound chime (`Rich Aesthetics`).
3. **True Two-Way Notion PARA API Property Sync:** When you tap `[+] Add Tag` inside the UI (`with just a name, icon and color`), our `NotionService` immediately calls the Notion API (`patch-database`) to append that new select option directly into your `Activity` property definition on your `Time Logs` Notion database. Your UI cards and your Notion schema stay 100% mirrored in real time without manual database editing.
4. **Zero-Breakage Encapsulation of Pomodoro Superpowers:** By mounting our new `Time Tracker` feature cleanly inside Tab 2 of a unified `AppView` navigation bar (`lib/pages/time_tracker/`) and renaming our launcher display name to **Time Tracker** (`com.recoskyler.timetracker`), all existing Pomodoro timers, floating desktop overlays (`OverlayApp`), and RGB webhooks continue operating untouched in Tab 1.

## Constraints
- **Strict Linting & State Rules:** Must pass `very_good_analysis` with zero suppressed lints unless technically justified. State mutations must use immutable `state.copyWith(...)` patterns inside `flutter_bloc` (`TimeTrackerCubit`).
- **Android 16 (`API 36` / Baklava) Battery Optimization:** On Android 16, background alarms and foreground services are subject to strict system battery throttling during Doze mode. The application must explicitly request `REQUEST_IGNORE_BATTERY_OPTIMIZATIONS` permissions (`Settings.ACTION_REQUEST_IGNORE_BATTERY_OPTIMIZATIONS`) and maintain an ongoing foreground notification (`notification.ongoing = true`) to ensure exact hourly beeps (`SCHEDULE_EXACT_ALARM`) fire reliably without being deferred by the OS.
- **Sleep Hour Suppression:** Must provide a configurable quiet-hours window inside Settings (`quietHoursStart` to `quietHoursEnd`, e.g., `23:00 to 07:00`). During this window, hourly background alarms and audio chimes must be automatically suppressed, and the hours must be marked as `Resting` rather than `Untracked / Missed` across all analytics views.
- **No Em Dashes:** Absolutely no em dash (`\u2014`) characters permitted anywhere in generated markdown, code strings, comments, commit messages, or user-facing text.

## Premises
1. **PREMISE 1 (Zero-Breakage Encapsulation & Existing Code Leverage):** We keep every line of existing `pomo` code (`TimerCubit`, `SettingsCubit`, `Prefs()`, `HookHelper`) untouched inside Tab 1 of a new navigation shell (`AppView`), building all new hourly grid and analytics features inside `lib/pages/time_tracker/` while updating our bundle launcher name to **Time Tracker** (`com.recoskyler.timetracker`).
2. **PREMISE 2 (Notion Dual-Database Schema & Real-Time API Tag Sync):** You configure two Notion database IDs (`PARA Projects` for active/inactive project modal + `Time Logs` for hourly rows). When you tap `[+] Add Tag` (`Name, Icon, Color`), `NotionService` immediately calls the Notion API (`patch-database` schema update) to append that new select option to your `Activity` property definition while saving it in local `Prefs()`.
3. **PREMISE 3 (Android Background Alarms & Sleep Exclusion Engine):** We use a native background alarm service (`android_alarm_manager_plus` / `flutter_local_notifications`) combined with Android 16 battery optimization bypass requests to sound an hourly beep on your Android phone. If the current time falls within your configured quiet hours (`SettingsState.quietHoursStart` to `quietHoursEnd`), the alarm suppresses the beep and automatically marks those hours as `Resting` on your 24-hour block timeline so they never count as missed tracking debt.
4. **PREMISE 4 (Distribution & Cross-Platform CI/CD):** Our automated build scripts (`build-web.sh`, `flutter build macos --release`, `flutter build apk`) will produce standalone distribution artifacts for all three targets (macOS `.dmg` / `.app`, Web PWA deployed to Vercel, and Android `.apk`), checked by `./scripts/verify.sh` on every pull request.

## Approaches Considered
### Approach A: Single-Package Modular Navigation & Direct API Sync (Minimal Viable)
- **Summary:** We keep all code inside our primary `pomo` `pubspec.yaml` (renamed to `Time Tracker`), adding a clean `NavigationBar` in `AppView` that hosts our untouched `TimerPage` in Tab 1 and our new `TimeTrackerPage` (4x5 grid + `fl_chart` analytics + missed hours block timeline) in Tab 2. Hourly Android beeps run via `android_alarm_manager_plus` checked against `quietHoursStart/End`, and Notion API calls (`create-page` and `patch-database` for custom UI tags) execute directly from our `NotionService`.
- **Effort:** Medium (~2 days)
- **Risk:** Low
- **Pros:**
  - Touched files and diff footprint are minimal, ensuring zero breakage to existing `TimerCubit` and `SettingsCubit` code.
  - Ships fastest to all three platforms (macOS `.app`, Android `.apk`, Web PWA) using our existing `scripts/build-web.sh` and `verify.sh` suite.
  - Direct Notion API calls make custom tag creation (`[+] Add Tag`) instantly visible in your Notion `Time Logs` database.
- **Cons:**
  - Direct network API calls during the top-of-the-hour check-in may fail or time out if your Android phone has a poor cellular connection during commute hours.
  - Does not include a dedicated local SQLite offline database queue, relying on `Prefs()` for temporary caching.

### Approach B: Clean-Architecture Feature Module with Background Worker & Desktop IPC (Ideal Architecture)
- **Summary:** We architect `lib/pages/time_tracker/` with strict clean architecture layers (`data/`, `domain/`, `presentation/`), building a `TimeTrackerRepository` that records all hourly check-ins to local disk first (`sqflite` / `Prefs`), while a background isolate (`TimeTrackerSyncWorker`) pushes entries and patches Notion database schemas resiliently with exponential backoff. On Android 16 (`API 36`), we request `REQUEST_IGNORE_BATTERY_OPTIMIZATIONS` permissions and maintain an ongoing persistent foreground notification (`notification.ongoing = true`) with interactive 1-tap action buttons. On desktop, we extend `DesktopWindowService` so our floating overlay (`OverlayApp`) displays an interactive "Log past 60m?" pill synchronized via IPC channels with our main `TimeTrackerCubit`.
- **Effort:** Large (~3-4 days)
- **Risk:** Low
- **Pros:**
  - Guarantees 100% offline resilience and zero-latency UI responsiveness on Android by saving every hourly check-in locally before syncing to Notion in the background with exponential backoff.
  - Delivers a jaw-dropping desktop companion experience where you can log past hours directly from your macOS menu bar or floating overlay without opening the main window.
  - Full compliance with Android 16 (`API 36`) battery throttling restrictions via explicit battery optimization bypass requests (`permission_handler`) and persistent foreground notification action hooks (`flutter_foreground_task`).
  - Strict domain layer separation makes testing and future expansion effortless without touching POMO code.
- **Cons:**
  - Requires creating more boilerplate classes (`repository interfaces`, `sync workers`, `IPC message handlers`) across our feature folder.
  - Takes slightly longer to build and test across our three deployment platforms.

### Approach C: Event-Driven PARA Automation Hub & SSE Live Sync (Creative / Lateral)
- **Summary:** We turn `Time Tracker` into an event-driven automation hub where `TimerCubit` lap events and hourly Android alarms emit domain events across a `ParaEventBus`. A `NotionParaEngine` pre-fills card suggestions ("Suggested: Work - 45m active in POMO"), while our Vercel API proxy (`deploy/focus/api`) receives Notion webhooks to push live tag updates down to your mobile app grid via Server-Sent Events (`SSE`) without polling.
- **Effort:** Extra Large (~5 days)
- **Risk:** Medium
- **Pros:**
  - Creates an intelligent bridge where your POMO focus sessions automatically credit towards your hourly PARA reports without manual typing.
  - Live SSE push notifications mean tags added in your master Notion desktop workspace instantly pop up on your mobile grid.
- **Cons:**
  - Requires deploying and maintaining custom webhook ingestion logic on our backend Vercel serverless proxy.
  - Higher complexity in managing persistent SSE connections across Android background sleep cycles.

## Recommended Approach
We choose **Approach B (Clean-Architecture Feature Module with Background Worker, Android 16 Battery Optimization Bypass, Persistent Ongoing Notification, and macOS Desktop IPC Bridge)**, consolidated into a minimal 7-file scope during `/plan-eng-review` (`Option B` Scope Challenge `D3`). This approach perfectly aligns with your stated goal (`"I want it to work for me first"` - Builder Mode Option B) and directly addresses your technical requirements for Android 16 (`API 36`). By combining a lightweight JSON `Prefs.pendingTimeLogs` offline retry queue (`D4`) with an ongoing persistent foreground notification (`notification.ongoing = true`) and defensive exception catching (`D5`), your Android phone will sound exact hourly beeps and allow 1-tap logging directly from the lock screen or notification drawer 24/7, without ever crashing or dropping offline data during Doze mode.

## Engineering Decisions (Eng Review 2026-07-13)
- **Scope Reduction (`D3`):** Consolidated `AnalyticsView` and `MissedHoursView` into a unified summary screen, and replaced `sqflite` database dependencies with a JSON list queue inside `Prefs()` (`SharedPreferences`).
- **Offline Resilience (`D4`):** Added `Prefs.pendingTimeLogs` JSON queue inside `NotionSyncService` to cache failed check-ins during cellular dropouts and auto-flush on reconnection with exponential backoff.
- **Android 16 `API 36` Crash Prevention (`D5`):** Wrapped `context.startForegroundService(intent)` in `TimerForegroundService.kt:38-43` inside a defensive `try/catch (e: Exception)` block with automatic fallback to `NotificationManager.notify()` to prevent `ForegroundServiceStartNotAllowedException` crashes during background check-ins.
- **Idempotency & Cumulative Task Time (`D6`):** Updated `NotionService.logSession()` (`lib/services/notion_service.dart:256-259`) so when `checkIdempotency(externalId)` matches an existing `Time Logs` row, step 3 and step 4 (`Task patch`) still execute, preventing parent task cumulative `Time (hours)` from drifting out of sync on retry.
- **Automated Regression Tests (`D7`):** Added two unit test groups inside `test/services/notion_service_test.dart` to verify `pendingTimeLogs` JSON caching and `checkIdempotency` cumulative time step jumping.
- **Network Latency Optimization (`D8`):** Added local cumulative delta math and a 5-minute TTL guard before step 3 (`dio.get`), cutting cellular HTTP roundtrips by 50% (`D8`).
- **Scalability Backlog (`D9`):** Created `TODOS.md` tracking future migration from `Prefs.pendingTimeLogs` to an indexed `sqflite` table if pending log accumulation exceeds 1,000 entries/month.
